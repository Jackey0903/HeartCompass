# Digital Immortality 项目简要介绍

2026.05.07

## 1. 一句话介绍

`Digital Immortality` 是一个以 **人物画像构建 + 持续对话** 为核心的 AI 系统。  
它不是普通聊天机器人，而是让你为“某个具体人物”沉淀结构化记忆，再让 AI 基于这些记忆进行更贴近该人物的对话输出。

它当前的主要交付形态不是 Web 页面，而是：

- `CLI`（命令行）作为管理和运维入口
- 飞书 Bot 作为日常会话入口

---

## 2. 这个项目要解决什么问题（Why）

很多“AI 人设/角色对话”系统都有共同问题：

- 人设信息散落在长文本里，模型每轮都全量读，成本高、噪音大
- 短期记忆无限增长，越聊越慢、越聊越飘
- 用户补充新信息后，历史画像难以增量修正
- 工程层面往往只停留在 prompt 文档，缺少可运行、可维护的系统化实现

`Digital Immortality` 的核心思路是把这件事工程化：

- 用结构化数据模型管理人物与关系
- 用向量召回按需拉取“与当前语境相关”的细粒度信息
- 用图工作流（LangGraph）把“抽取、对齐、更新、回复”变成稳定流程
- 用 CLI + 飞书把能力暴露为可持续运行的产品形态

---

## 3. 产品层能做什么（What）

### 3.1 两条核心链路

1. **人物画像完善链路（FRBuildingGraph）**

- 输入一段原始材料（例如你对某人的描述）
- 系统抽取信息并更新人物核心字段
- 同时写入细粒度信息库（按维度拆分）
- 生成本轮“画像完善报告”

2. **对话生成链路（ConversationGraph）**

- 输入本轮用户消息
- 结合人物画像 + 相关记忆召回 + 短期记忆
- 产出该人物本轮回复 `messages_to_send`
- 自动做短期记忆 trim 和滚动摘要，防止上下文失控

### 3.2 用户入口

- `CLI`：注册登录、环境检查、人物管理、画像查看、feeds 同步、飞书服务启动
- 飞书 Bot：切换对话对象、查看画像、补充画像、日常对话

---

## 4. 系统总体架构（How）

可以把系统理解为 5 层：

1. **Channel 层**

- 飞书 WebSocket 接入与消息收发

2. **CLI 层**

- 命令树入口，负责本地配置、健康检查、用户操作

3. **Agent/Graph 层**

- `FRBuildingGraph`（画像构建）
- `ConversationGraph`（对话生成）
- Checkpointer（短期记忆落库与恢复）

4. **Service 层**

- 用户、人物关系、细粒度信息等业务操作入口
- 对上层暴露稳定函数协议

5. **数据层**

- PostgreSQL + pgvector
- 关系数据 + 向量检索 + 冲突与更新日志

一句话：**Graph 负责智能流程编排，Service 负责业务与数据读写，CLI/飞书负责把能力交给用户。**

---

## 5. 关键业务对象（数据建模）

项目的“灵魂”是这几个数据对象：

- `User`：用户主体，包含认证信息和飞书 `open_id` 绑定
- `FigureAndRelation`（FR）：某个用户视角下的“人物+关系”主实体
- `OriginalSource`：用户本次输入的原始材料（预处理后）
- `FineGrainedFeed`：细粒度画像信息（可向量召回）
- `FineGrainedFeedConflict`：冲突信息记录（待人工/流程后续处理）
- `FROverallUpdateLog`：FR 字段或 feed 变动日志
- `FRBuildingGraphReport`：画像构建的可读报告

`FineGrainedFeed` 当前按维度组织：

- `PERSONALITY`（性格与价值观）
- `INTERACTION_STYLE`（互动风格）
- `PROCEDURAL_INFO`（做事方式/程序性信息）
- `MEMORY`（人生记忆与故事）

这让系统可以“按需召回”而不是“全量加载”。

---

## 6. 画像构建链路详解（FRBuildingGraph）

当用户说“请补充这个人的画像”时，系统做了什么？

### 6.1 流程主干

1. 加载并校验当前 FR（用户归属、角色等）
2. 预处理原始输入（清洗内容、识别来源类型、置信度、涉及维度）
3. 落库 `OriginalSource`
4. 并行执行两条分支：

- 分支 A：抽取 FR 内在字段候选 -> 规划更新 -> 落库更新
- 分支 B：抽取细粒度 feeds -> 规划 upsert -> 落库（新增/更新/冲突记录）

5. 汇总本轮结果
6. 生成并落库“画像完善报告”

### 6.2 设计重点

- **不是盲目覆盖**：先比较旧值和新值，再决定更新策略
- **冲突可追踪**：发现冲突会落冲突表，不是直接吞掉
- **日志可观测**：每个节点写 `logs/warnings/errors`，便于排查
- **并发受控**：`FRBuildingGraph` 通过 `Semaphore(1)` 限制同一时刻仅一个任务运行

---

## 7. 对话链路详解（ConversationGraph）

当飞书里收到用户消息后，系统如何回复？

### 7.1 对话入口策略

飞书消息不是“来一条处理一条”，而是：

- 先按用户缓冲消息
- 等待 `WAITING_SECONDS_FOR_CONVERSATION`（默认 15 秒）
- 若期间又来新消息，重置计时器
- 计时结束后把这一批消息一起送入 `ConversationGraph`

这更接近真实聊天节奏，也降低每条消息都跑一次图的成本。

### 7.2 Graph 核心节点

1. `nodeLoadFRAndPersona`

- 加载用户和 FR
- 构建人物画像 markdown（部分字段排除，避免重复注入）

2. 并行分支

- `nodeRecallFeedsFromDB`：按当前消息语义召回高语境依赖信息（目前重点是 `MEMORY` 和 `PROCEDURAL_INFO`）
- `nodeBuildAndTrimMessage`：追加本轮 `HumanMessage`，并进行短期记忆裁剪

3. `nodeCallLLM`

- 组装系统提示词 + 人物画像 + 召回结果 + 历史摘要 + 当前消息
- 调用模型生成 `messages_to_send`
- 回写 `AIMessage` 进入短期记忆

### 7.3 短期记忆策略（非常关键）

项目不是简单“越聊越堆消息”，而是：

- 设定字符数和消息数阈值
- 超阈后按 `round_uuid` **整轮裁剪**（尽量不拆开一轮的人类输入和 AI 回复）
- 被裁掉的历史消息不丢弃，转入滚动摘要 `conversation_summary`
- 后续作为系统信息注入，保留长期语义连续性

这解决了长期会话的典型问题：上下文爆炸、性能下降、回复漂移。

## 8. CLI 设计与运维体验

项目把 CLI 作为一等公民，主要体现在：

- 统一命令入口：`doctor / setup / auth / fr / lark-service / logs`
- 支持 `--json` 输出，便于 Agent 和脚本消费
- 本地状态与配置统一放在 `~/.immortality/`

### 8.1 首次使用体验

1. `immortality setup`

- 可选 Docker 自动拉起 PostgreSQL + checkpoint 数据库
- 生成本地 `.env`
- 初始化数据库表和向量扩展

2. `immortality doctor`

- 检查 Python 版本、必填环境变量、依赖完整性、数据库连通性

3. `immortality auth login`

- 建立本地 session

4. `immortality lark-service start`

- 再次自动做 doctor，通过后启动飞书长连接服务

---

## 9. 飞书集成是怎么落地的

飞书侧主要做了三件事：

1. **消息入口**

- 通过长连接事件接收文本消息

2. **命令菜单**

- 例如 `/menu`、`/list_available_persons`、`/<fr_id>`、`/show_persona`、`/build_persona`

3. **消息发送**

- 文本和卡片输出，反馈错误、成功、报告等状态

系统会把飞书用户 `open_id` 映射到本地用户，再校验当前 FR 归属，避免跨用户错用人物数据。

## 10. 技术选型的现实取舍

### 10.1 为什么是 Graph，不是纯 ReAct Agent

- 这个场景流程固定、步骤清晰（预处理、抽取、比较、落库、汇总）
- 需要稳定中间产物和节点级日志
- 需要可控并行、可控错误处理、可追踪状态

Graph 在这里比“完全自主决策型 agent”更可控、更适合工程交付。

### 10.2 为什么用结构化数据 + 向量召回

- 全量长文喂模型的方式，随着信息增长会迅速失效
- 结构化字段保证“基础画像稳定”
- 细粒度 feed + 向量召回保证“按语境取用相关信息”

两者组合，兼顾准确性和性能。

## 11. Harness 落地：把能力沉淀为可执行资产

在这个项目中，Graph 解决“智能流程编排”，Harness 解决“研发能力复用与稳定执行”。  
两者结合，系统才真正具备长期演进能力。

### 11.1 Harness 在本项目中的定位

Harness 的核心载体是 `skills` 资产。每个 skill 都是一个可执行能力单元，明确约束：

- 触发时机：什么请求会路由到该能力
- 输入契约：执行前必须具备哪些上下文
- 执行步骤：在仓库内如何完成任务
- 质量门槛：输出达到什么标准才算完成
- 禁止事项：哪些行为会导致流程失真或失控

它的价值在于把“个人经验”沉淀为“团队可复用规程”，并接入实际研发链路。

### 11.2 当前已落地的 Skill 资产

项目当前主要 skill 包括：

- `commit-quality-reviewer`
- `commit-update-writer`
- `doc-generator`
- `doc-optimizer`
- `graph-doc-writer`
- `session-practice-learner`

这些能力覆盖了改动质检、更新记录、文档生成优化、Graph 文档对齐、会话经验沉淀等高频环节。

### 11.3 Skill 的生产方式（从实践到资产）

本项目采用“从真实高频任务反推资产”的方式沉淀 skill：

1. 识别稳定且可复用的高频动作
2. 定义触发条件、输入约定、执行流程、验收标准
3. 在实战中消费并观测结果
4. 对偏差直接回写规则而不是只修当次结果
5. 在后续任务复用验证，形成闭环

### 11.4 Agent 如何消费 Harness

在日常协作中，Agent 会先做意图路由，再执行对应 skill：

- 改动审查 -> `commit-quality-reviewer`
- 改动文档 -> `commit-update-writer`
- 文档成文 -> `doc-generator`（并衔接 `doc-optimizer`）
- Graph 文档 -> `graph-doc-writer`
- session 收尾沉淀 -> `session-practice-learner`

关键点是严格遵循 skill 的输入/输出契约，保证结果稳定可复现。

### 11.5 一个完整闭环示例

一次典型研发链路是：

1. 完成代码改动
2. 触发差异质检并修复阻塞问题
3. 生成并追加更新记录
4. 如涉及文档，执行成文与优化
5. session 收尾判断是否沉淀新资产

这条链路的重点是：每次执行不仅产出当次结果，也持续反哺 Harness 资产本身。

## 12. FAQs

### Q1：和市面上的角色扮演聊天产品有什么区别？

区别在“数据组织方式”和“工程可持续性”：

- 不是只靠一段人设 prompt
- 有结构化画像、细粒度记忆、向量召回、可追踪更新日志
- 可通过 CLI 和飞书长期运行

### Q2：为什么没有先做前端 App？

项目优先解决“能力闭环和稳定性”，先用 CLI + IM Channel 快速验证真实使用链路，降低交付成本。

### Q3：会不会越聊越乱？

系统有短期记忆 trim + 滚动摘要机制，专门防止上下文无限膨胀和语义漂移。

### Q4：如果抽取错了怎么办？

画像链路有比对和冲突机制，更新过程有日志可追踪，后续也在加强高风险写入的人机确认。
