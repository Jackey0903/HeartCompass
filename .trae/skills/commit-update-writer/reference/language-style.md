# Immortality 文风参考（强制）

本文件用于约束 `commit-update-writer` 的写作风格。  
每次产出更新记录前，必须先阅读本文件。

## 参考来源

- `docs/REFACTOR.md`
- `docs/BOTTLENECK.md`
- `docs/HEARTCOMPASS.md`

## 文风特征（必须遵守）

1. 先讲“为什么”，再讲“怎么做”
- 先交代背景、目标、约束，再进入技术实现。
- 不要一上来就贴技术细节。

2. 叙事要真实，允许保留思考与取舍
- 可以写“我尝试了什么、为什么没走通、最后怎么决策”。
- 对未解决问题要直说，不包装成“已完成”。

3. 工程细节要具体可追溯
- 使用真实模块名、函数名、环境变量名、命令名（用反引号）。
- 避免空泛表达，如“做了优化”“提升了性能”但无依据。

4. 中文为主，夹带必要英文术语
- 保持中文叙述流畅，英文术语用于精确表达（如 `checkpointer`、`dispatch`、`thread_id`）。
- 英文术语和代码符号统一用反引号。

5. 强调链路视角，而不是孤立改动
- 说明“上游依赖 -> 当前改动 -> 下游影响”。
- 明确哪些环节已验证，哪些环节仍有风险。

6. 结构清晰但不过度模板化
- 推荐分段：背景与目标、改动动作、关键问题、结论与后续。
- 可使用编号和短列表，但不要堆砌无意义小标题。

## 语言表达风格（必须遵守）

1. 语气：工程复盘式，而不是公告式
- 推荐用“我/我们”叙述真实决策过程，语气克制、具体。
- 避免“本次全面升级”“显著提升”等空泛宣传语。

2. 句式：短句为主，关键处允许中长句解释因果
- 每段优先表达一个核心观点。
- 复杂因果链可用“因为...所以...；但...因此...”展开。

3. 过渡：强调问题驱动与决策转折
- 常用连接词：`一开始`、`但`、`因此`、`换句话说`、`最终`、`接下来`。
- 过渡必须服务于逻辑，不要为“文采”堆砌表达。

4. 术语密度：可读优先，精确兜底
- 先用中文解释，再补必要英文术语（放在反引号中）。
- 单段避免过多术语堆叠，避免读者失去主线。

5. 风险表述：避免绝对化结论
- 使用“当前看起来”“在这个边界下”“仍有风险点”等审慎表述。
- 明确“已验证 / 未验证 / 计划验证”，不要用“完全解决”。

## 表达示例（建议直接参考）

### 示例 1：背景与目标（推荐写法）

```markdown
这次改动的起点不是“功能扩展”，而是一个稳定性问题：`ConversationGraph` 在长对话下上下文体积持续膨胀，导致响应时间波动明显。  
一开始我尝试只做消息条数裁剪，但很快发现这个方案会破坏轮次语义（同一轮的人类输入和 AI 回复被拆开）。因此这次我把目标改成“按轮裁剪 + 滚动摘要”，优先保证语义完整性，再控制体积。
```

### 示例 2：链路解释（推荐写法）

```markdown
上游依赖是 `messages` 的增量合并语义（`MessagesState`），这决定了 trim 不能用整表覆盖。  
当前改动在 `nodeBuildAndTrimMessage` 中返回 patch：删除旧消息使用 `RemoveMessage`，新增输入只追加本轮 `HumanMessage`。  
下游影响是 `nodeCallLLM` 消费到的上下文更稳定；但代价是触发 trim 时会新增一次摘要模型调用，延迟会有小幅上升。
```

### 示例 3：风险与边界（推荐写法）

```markdown
这次改动解决了“短期记忆无限增长”的主问题，但没有彻底解决所有性能波动。  
原因是 `nodeCallLLM` 仍然每轮拉取 prompt，且日志里保留了较大的调试输出。换句话说，当前收益主要来自记忆体积控制，而不是全链路 I/O 优化。  
接下来我会先补两类验证：一类是 50 轮对话下的延迟分布，一类是 trim 触发频率与摘要质量的相关性。
```

## 反例（避免这样写）

```markdown
我们对系统进行了大量优化，性能得到显著提升，稳定性完全没有问题。  
本次改动涉及多个模块，整体效果非常好。
```

问题：
- 没有交代“为什么改”。
- 没有指出“改了什么链路”。
- 没有给出任何边界与风险，表达失真。

## 禁止写法

- 只贴 diff，不解释业务目的。
- 只写“做了什么”，不写“为什么这样做”。
- 回避失败尝试与风险，导致记录失真。
- 用“完美解决”“完全无风险”等绝对化措辞。
