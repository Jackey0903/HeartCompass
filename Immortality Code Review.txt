会议纪要
视频主要对代码逻辑和架构进行审查，介绍了代码架构的分层、各层功能及特点，探讨了函数式编程风格在项目中的应用，参会人员对项目给予肯定并提出疑问和建议，内容如下：​
代码架构介绍​
架构分层​
分层概述：代码架构分为 channel 层、CLI 层、agent 层、service 和数据层，编排参考依赖倒置原则，每层预留接口，依赖的能力由其他部分暴露。​
目录结构：业务工程代码在 source 目录下，外部资源如测试、脚本、提示词等在其他目录，AI Agent 相关内容在 agents 目录。​
各层功能​
AI 层：暴露 SDK、外部记忆库、模型实例等，采用函数式编程风格，封装 toolcall 类用于参数处理，工具调用自定义函数。​
API 层：封装 API 以满足共享数据库等需求，利用 service 层能力，通过 RESTful API 暴露服务。​
Service 层：作为业务逻辑的唯一出口，分为 user knowledge FR 和 feed 等几类，提供基本 CRUD 方法和业务相关方法。​
Database 层：负责数据持久化，包括数据表定义、数据库 ORM 连接、数据类型定义等，利用 Alembic 工具同步数据库修改。​
Utils 层：包含工具函数，如 JSON 解析、文本化处理等。​
CLI 层：面向用户的交付层，包含多层命令，有子参数，通过 main 作为唯一入口，用户登录凭证等信息存储在本地配置文件。​
Channel 层：目前仅支持飞书，借助飞书 SDK 启动服务，提供指令化功能，消费 service 层能力。​
Harness 落地流程​
技能沉淀：将开发过程中沉淀的 Skill 资产加载到 agent 中，如 Commit Quality Reviewer 用于质检。​
白名单管理：对质检发现的问题进行评估，将误报问题加入白名单，后续检查不再考虑。​
Change Log 生成：根据代码改动生成 change log，记录关键链路、改动结果、风险等信息，并生成 commit message。​
规则约束：加载全局规则，约束 AI 生成内容，避免重复表述。​
参会人员意见与建议​
代码风格​
个人习惯：卜天表示采用函数式编程是个人习惯，开发 Java 久的人可能习惯面向对象，开发 TypeScript 久的人更习惯函数式。​
多人协作：李冉曦提出 OOP 在多人协作上可能有优势，卜天认为函数式和 OOP 在上下游依赖问题上本质相同，改动时需考虑下游使用情况，可通过增量式改动和版本更新解决。​
